Option D — Entity Scope vs Entity Tag

Prototype demonstrating the two-mechanism model. Atomic cards carry an optional entity tag (metadata only — drives search, no behavior). Compounds carry a required entity scope (behavioral — wires constituents together).

entity tag — metadata only entity scope — behavioral atomic card compound wired constituent

Entity Tag (atomic cards)

  • Optional — card works without it
  • Pure metadata; drives search and file browser filtering
  • Has no onEntityChange handler by type design
  • Multiple entities can be tagged on one card
  • Cannot be read by siblings — no shared state

Entity Scope (compounds)

  • Required — compound cannot exist without a scope entity
  • Drives constituent behavior: map bounds, timeline range, text seed
  • Scoped state is readable by all constituents in the compound
  • One scope entity per compound (multi-entity = multiple compounds)
  • Constituents may carry their own tag independently

Key constraint: Atomic card types have no onEntityChange handler defined at the type level. Behavioral attachment is structurally impossible — not just discouraged by convention. A compound's scope entity does have handlers, but they are only accessible inside the compound runtime and are not exposed to the card API.
